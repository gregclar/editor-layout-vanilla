Name search

Names listed in alphabetical order.

Common and cultivar names are excluded by default.

Quick start for Name search

Type the first character or word in the name.

Start the search by pressing Enter or clicking the "Search" button....

or leave a space then enter another fragment of the name.

Start the search.

The Editor searches very broadly - it adds a trailing wildcard and also treats spaces as wildcards.

It does not add a leading wildcard to name search terms - to keep searches as fast as possible.

E.g. grev queen, bank spin or e costata

Order of terms is important for name searches

Whatever fragments you enter need to be in the order you expect them in the name or names.

If you do not know how the name starts, lead with a wildcard ("*"). E.g. *queensland form

But this search should be faster: g*queensland form.

Common and Cultivars in Name searches

Name searches may or may not include common and cultivar names.

There is a toggle to set this preference - either On or Off.

When set Off, common and cultivar names will be excluded from the search unless your search requires them to be included.

Searches by record ID or searches that specify a name type which includes common or cultivars ("type:*","type:cultivar") are examples that require common and cultivars to be included.

When set On, common and cultivar names will be included in the search unless your search excludes them from the result.

A search that specifies "nt:scientific" is an example that would exclude common and cultivars even if the setting is On.

Showing Instances

A successful Name search returns a list of names by default.

You can ask for Instances as well as Names in the results by adding the show-instances: directive to the search.

This directive takes no argument.

The show-instances: directive also has abbreviations: s-i:, si:, and i: that all do the same thing.

Case and Wildcards in Searches

Editor searches are all case-insensitive.

Editor searches on text fields allow you to add wildcards. e.g. "*"

Fields for which the Editor adds its own wildcards are noted and an "-exact" version of the field is provided that adds no wildcards.

You can add your own wildcards to "-exact" fields.

Steps
  1. Query and view loader-name records to see how well they match with existing name records.
    • You can edit the loader-name simple_name to fine-tune the set of matching names (original value is retained in simple_name_as_loaded)
    • Create name records if no match exists
    • Delete loader-name records if not required
    • Exclude a loader-name record from further processing at any time (reversible)
    • Create new raw loader-name records via New > Loader Name to supplement the import
    • Ensure any potential matching Name records have a primary instance
    • View misapplication matches to a name broken down to specific instances - to let you see and choose where the misapplication occurred
  2. Set preferred match(es) for loader-names - locking them to the appropriate name record
    • Edit the match if you need to adjust the relationship instance
  3. Identify an instance that documents the usage of the loader-name for the taxonomy.
  4. If no such instance exists, create one.
  5. Add the identified instance to the draft taxonomy.
